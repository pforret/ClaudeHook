2026 05 21.claudehook say and install

name: "ClaudeHook say + install — voice notifications wired into Claude Code hooks" description: |

Purpose¶

Add two new verbs to ClaudeHook.sh so the tool becomes useful end-to-end:

  1. ClaudeHook say <whatever> — speaks "<APP_NAME> <whatever>" out loud (macOS say / Linux spd-say/espeak, echo fallback). APP_NAME is taken from .env if present, otherwise from basename $PWD.
  2. ClaudeHook install — interactive installer that (a) asks global (~/.claude/settings.json) vs project only (./.claude/settings.json), then (b) prompts the user to install five Claude Code hooks with sensible defaults that each call back into ClaudeHook say …: Notification, Stop, StopFailure, PreCompact, PermissionRequest.

The result: after one ClaudeHook install, the user hears their machine speak the project name whenever Claude needs attention, finishes, fails, compacts, or asks for permission.

Core Principles¶

  1. Match bashew conventions — add verbs to Option:config() choice line, add case blocks in Script:main(), implement do_<verb>() helpers, use IO:* for output and Os:require for binaries.
  2. No new abstractions — this is a single-file bash script. No new files except optional tests. Keep functions short, no premature library.
  3. Idempotent installs — re-running install must not duplicate hook entries in settings.json. Use jq for safe JSON merges.
  4. Testable in CI without speaking — fall back to echo when no TTS binary is present (Linux CI has none), and provide a --DRY_RUN flag so a developer on macOS can verify the spoken text without audio.

Goal¶

After this PRP is implemented, the following user journeys work end-to-end:

# 1) Speak a custom message (used by the hooks, but also directly callable)
$ cd ~/Code/myproject
$ ClaudeHook say "is done"
# → speaks: "myproject is done"   (basename $PWD when no .env)

$ echo 'APP_NAME=Hermes' > .env
$ ClaudeHook say "needs your attention"
# → speaks: "Hermes needs your attention"

# 2) Install hooks interactively
$ ClaudeHook install
Install hooks for: [1] this project only  [2] global (all projects) > 2
Install Notification hook?       (will say "<app> needs your attention") [y/N] y
Install Stop hook?               (will say "<app> is done") [y/N] y
Install StopFailure hook?        (will say "<app> encountered an error") [y/N] y
Install PreCompact hook?         (will say "<app> is compacting context") [y/N] n
Install PermissionRequest hook?  (will say "<app> needs permission") [y/N] y
✅ 4 hooks installed in /Users/pforret/.claude/settings.json

# 3) Force mode bypasses prompts and installs all hooks globally
$ ClaudeHook -f install
✅ 5 hooks installed in /Users/pforret/.claude/settings.json

Why¶

  • The bashew template in ClaudeHook.sh currently has only placeholder verbs (action1, action2). The repo name promises Claude-Code-hook functionality but the script does nothing useful yet.
  • Voice notifications for hook events is a high-value, low-complexity Claude Code use case that this tool can own.
  • One install command is dramatically easier than hand-editing JSON in ~/.claude/settings.json.

What¶

User-visible behavior¶

ClaudeHook say [message] - Resolves the app name in this order: $APP_NAME (loaded automatically by bashew's Os:import_env from .env) → basename "$PWD". - Speaks "<app_name> <message>" using the first available of: say (macOS), spd-say, espeak (Linux). If none are available or --DRY_RUN is set, prints the message to stdout instead. - Empty message is allowed (just speaks the app name). Useful for testing. - Multi-word messages must be quoted on the CLI (consistent with bashew's param|? single optional positional).

ClaudeHook install - Asks: project (./.claude/settings.json) vs global (~/.claude/settings.json). Uses IO:question with a default of project. - For each of the 5 hooks, prompts y/N via IO:confirm. With -f/--FORCE all prompts auto-yes (this is the documented bashew IO:confirm behavior). - The command written into settings.json is the absolute path to ClaudeHook.sh (resolved via $script_install_path, already computed by Script:meta), so the hook works regardless of $PATH. - Hook entries are merged into existing settings.json via jq. Re-running does not create duplicates (the implementation removes any existing entry with an identical command string before appending).

Success Criteria¶

  • ClaudeHook say "hello" in a directory with no .env says " hello" (or prints it on Linux without TTS).
  • ClaudeHook say "hello" in a directory whose .env contains APP_NAME=Foo says "Foo hello".
  • ClaudeHook install (interactive) creates a valid ~/.claude/settings.json or ./.claude/settings.json containing the chosen hooks under .hooks.<EventName>[].
  • Each installed hook entry has shape {"type": "command", "command": "<abs-path>/ClaudeHook.sh say \"<message>\"", "timeout": 10}.
  • Running ClaudeHook -f install twice in a row results in the same final JSON (idempotent — no duplicate entries).
  • The resulting settings.json is valid JSON (jq . settings.json succeeds).
  • shellcheck ClaudeHook.sh produces no new warnings.
  • bash_unit tests/test_* passes, including new tests for say and install.
  • Help output (ClaudeHook --help) lists say and install as actions, and the #TIP: lines render in the tips section.

Known Gotchas of our codebase & Library Quirks¶

# CRITICAL: bashew strict mode is `set -uo pipefail` (no -e). Trap on EXIT runs IO:die on any
# non-zero exit from a *piped* command. Read ClaudeHook.sh line 195 and 496–499.
# → Inside do_install(), prefer `if ! cmd; then …; fi` over relying on $? after a pipeline.

# CRITICAL: Os:import_env (line 1062) runs BEFORE Option:parse and BEFORE Script:main.
# So $APP_NAME from .env is already in the environment when do_say() runs. Don't re-source.

# CRITICAL: bashew uses `param|?|input|input file/text` (line 90). $input is a single optional
# positional. Multi-word messages MUST be quoted by the caller:
#   ClaudeHook say "needs attention"     ← correct
#   ClaudeHook say needs attention       ← bashew dies with "cannot interpret extra parameters"
# This is intentional — hooks always pass a quoted message, so it's the natural shape.

# CRITICAL: $script_install_path (set in Script:meta, line 952–957) is the symlink-resolved
# absolute path to ClaudeHook.sh. Use this for the hook `command` field — NOT $0, which could
# be a relative path or the symlink "./ClaudeHook".

# CRITICAL: IO:confirm (line 317) returns 0 on Y, 1 on N. With -f/--FORCE it auto-returns 0.
# This means `ClaudeHook -f install` should install ALL hooks without prompting. Good for CI.

# CRITICAL: jq is not always installed. Os:require "jq" (with no second arg) will suggest
# `brew install jq` on macOS and `apt install jq` on Debian/Ubuntu via $install_package.
# Don't try to hand-roll JSON editing with sed — settings.json may have unrelated keys we
# must preserve verbatim.

# GOTCHA: The macOS `say` command speaks asynchronously and returns immediately. The Linux
# `spd-say` does too. `espeak` is synchronous. For hooks we want non-blocking so Claude
# doesn't wait — backgrounding (`&`) is safer cross-platform but interferes with our test
# harness. Compromise: run the TTS binary in the foreground; users on espeak can add `&` in
# their settings.json command if they want.

# GOTCHA: On the bashew CI runner (Linux, no TTS), `command -v say` and `command -v spd-say`
# both return empty. The fallback `echo` path is what tests exercise. Keep that branch clean.

# GOTCHA: When writing to ~/.claude/settings.json, create the parent dir if it doesn't exist
# (`mkdir -p`). For project scope, write to ./.claude/settings.json (note: the directory is
# .claude, not claude — and it must be created if absent).

Per task pseudocode¶

Task 3 — do_say body: 

function do_say() {
  IO:log "say: '$input'"

  local app_name
  if [[ -n "${APP_NAME:-}" ]]; then
    app_name="$APP_NAME"
  else
    app_name="$(basename "$PWD")"
  fi

  local full_message
  if [[ -n "${input:-}" ]]; then
    full_message="$app_name $input"
  else
    full_message="$app_name"
  fi

  # DRY_RUN or no TTS available → just print.
  if ((DRY_RUN)); then
    IO:print "$full_message"
    return 0
  fi

  if command -v say >/dev/null 2>&1; then
    say "$full_message"
  elif command -v spd-say >/dev/null 2>&1; then
    spd-say "$full_message"
  elif command -v espeak >/dev/null 2>&1; then
    espeak "$full_message" 2>/dev/null
  else
    # No TTS — degrade to stdout so the hook still produces visible feedback.
    IO:print "$full_message"
  fi
}

Task 5 — do_install body: 

function do_install() {
  IO:log "install"
  Os:require "jq"

  # 1) Resolve scope (allow CLI override via $input).
  local scope target_file target_dir
  case "${input:-}" in
    g|global|G|GLOBAL) scope="global" ;;
    p|project|P|PROJECT) scope="project" ;;
    "")
      local answer
      answer="$(IO:question "Install hooks for: [p]roject only / [g]lobal" "p")"
      case "${answer,,}" in
        g|global|2) scope="global" ;;
        *)          scope="project" ;;
      esac
      ;;
    *) IO:die "unknown scope [$input] — use 'global' or 'project'" ;;
  esac

  if [[ "$scope" == "global" ]]; then
    target_dir="$HOME/.claude"
  else
    target_dir="./.claude"
  fi
  target_file="$target_dir/settings.json"
  mkdir -p "$target_dir"

  # 2) Seed empty settings file if absent.
  if [[ ! -f "$target_file" ]]; then
    echo '{"hooks":{}}' > "$target_file"
    IO:debug "Created empty $target_file"
  fi

  # 3) Loop over hook events.
  local hook_events=("Notification" "Stop" "StopFailure" "PreCompact" "PermissionRequest")
  local hook_messages=("needs your attention" "is done" "encountered an error" "is compacting context" "needs permission")
  local installed=0
  local i event message cmd

  for i in "${!hook_events[@]}"; do
    event="${hook_events[$i]}"
    message="${hook_messages[$i]}"
    cmd="$script_install_path say \"$message\""

    if IO:confirm "Install $event hook? (will say \"<app> $message\")"; then
      # Merge into .hooks.$event, deduping by .command.
      local tmpfile
      tmpfile="$(Os:tempfile json)"
      jq \
        --arg event "$event" \
        --arg cmd "$cmd" \
        --argjson timeout 10 \
        '
          .hooks //= {} |
          .hooks[$event] = ((.hooks[$event] // []) | map(select(.command != $cmd)))
                          + [{type:"command", command:$cmd, timeout:$timeout}]
        ' \
        "$target_file" > "$tmpfile" || IO:die "jq failed updating $target_file"
      mv "$tmpfile" "$target_file"
      installed=$((installed + 1))
      IO:debug "Installed $event → $cmd"
    fi
  done

  # 4) Final validate.
  if jq -e . "$target_file" >/dev/null 2>&1; then
    IO:success "$installed hook(s) installed in $target_file"
  else
    IO:die "settings.json is not valid JSON after edits: $target_file"
  fi
}

Integration Points¶

SETTINGS.JSON (Claude Code):
  - global: $HOME/.claude/settings.json
  - project: ./.claude/settings.json
  - schema: {"hooks": {"<EventName>": [{"type":"command","command":"…","timeout":10}]}}
  - dedup key: the `.command` string (so re-runs are idempotent)

ENV:
  - APP_NAME — read from .env (auto-loaded by bashew Os:import_env). Falls back to basename $PWD.
  - CLAUDE_PROJECT_DIR — Claude Code sets this; basename $PWD inside the hook context is
    equivalent and simpler. We do NOT need to read this explicitly.

OPTION:CONFIG (bashew):
  - choice|1|action|action to perform|say,install,check,env,update
  - flag|D|DRY_RUN|print message instead of speaking (for testing)
  - param|?|input|input file/text  ← already exists; reused for `say <msg>` and `install <scope>`

Anti-Patterns to Avoid¶

  • ❌ Don't introduce a separate lib/ folder or split into multiple files — bashew's whole value is a single self-contained script.
  • ❌ Don't hand-edit JSON with sed/awk — use jq and require it explicitly. The dedup logic is impossible to do safely without a JSON parser.
  • ❌ Don't background the TTS command (say "$msg" &) inside do_say — let users decide via their own settings.json if they want non-blocking.
  • ❌ Don't read $CLAUDE_PROJECT_DIR to compute the app name — basename $PWD and .env already cover both interactive and hook-invoked contexts, and the cwd is what bashew's env loader keys off.
  • ❌ Don't keep action1/action2 "for backwards compatibility" — the script has zero users yet (initial commit was 1 day ago) and the help text is part of the deliverable.
  • ❌ Don't add a --global / --project flag in addition to the positional install global argument — pick one. The positional is simpler and reuses existing $input.
  • ❌ Don't write to ~/.config/claude/... or any other path — the Claude Code docs are explicit that the file is ~/.claude/settings.json.
  • ❌ Don't try to install hooks by appending to a file (>>) — that produces invalid JSON the moment there's a second hook.
